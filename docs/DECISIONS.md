# Decisions

## 2026-06-30：V1 使用 API-only 只读路线

决定：V1 只通过蓝湖 Web API 读取项目和设计稿信息，不做浏览器自动化兜底。

原因：

- 浏览器拖拽画布、截图和 OCR 路径笨重且不稳定。
- API-only 更适合 MCP 工具化，也更容易测试。
- 失败时返回结构化错误和建议，比隐式浏览器兜底更可控。

影响：

- 遇到 403/418 或接口字段变化时，V1 不自动打开浏览器补救。
- 后续优先增强 API 适配和错误提示。

## 2026-06-30：认证只使用 LANHU_COOKIE 环境变量

决定：V1 只读取 `LANHU_COOKIE` 环境变量。

原因：

- 不读取浏览器 Cookie、LocalStorage 或用户私密配置。
- 不保存、不打印 Cookie，安全边界清晰。
- 方便 Codex、Claude、Cursor 等 MCP 客户端配置。

影响：

- 新用户需要手动从蓝湖请求中复制 Cookie。
- Cookie 过期时需要手动更新。

## 2026-06-30：项目内私有 MCP 包

决定：当前阶段不发布 npm 包，先作为项目内私有工具开发。

原因：

- 先验证蓝湖 API 和 Codex 工作流。
- 减少发布、版本兼容和用户支持成本。
- 更适合快速迭代。

影响：

- 使用者需要 clone 仓库、安装依赖、构建后接入 MCP。
- 后续若多人稳定使用，再规划 npm 发布。

## 2026-06-30：多 Agent 协作靠仓库上下文承接

决定：不同 AI Agent 不依赖聊天记录共享上下文，而是依赖仓库文档和 git 历史接力。

原因：

- 多端、多用户、多 AI 客户端之间无法天然共享对话上下文。
- 文档化的项目记忆更稳定，也更适合代码审查和长期维护。

影响：

- 每个阶段必须更新 `STATUS`、`HANDOFF`，必要时更新 `ROADMAP` 和 `DECISIONS`。
- 新 Agent 必须先读 `AGENTS.md` 和 docs 后再开发。

## 2026-07-01：V1.1 优先建设代码还原上下文

决定：下一阶段优先把 `context.md/context.json` 从画板清单升级为辅助 Codex 还原代码的实现上下文。

原因：

- 项目核心目标是辅助 Codex 根据蓝湖设计文稿进行页面/组件代码还原。
- V1 已证明 API-only 读取链路可行，但输出还不足以指导高质量实现。
- 在尚未获得完整图层树、切图和 Design Tokens 前，可以先通过画板名称、资源路径和轻量规则生成页面角色、流程关系和实现建议。

影响：

- 新增 `src/core/restoration.ts` 作为还原推断层，避免把推断逻辑散落在 MCP 工具层或 Markdown 渲染层。
- `context.json` 新增 `restoration` 结构，后续单图详情、切图和 tokens 都应接入该结构或其后续扩展。
- 页面角色和流程推断必须明确标注为轻量推断，不得伪装成蓝湖官方结构化图层数据。

## 2026-07-01：新建议必须围绕第一原则沉淀

决定：后续每次出现新的实现建议、路线建议或协作建议时，必须先按第一原则评估；不冲突且有价值的建议应写入项目文档。

第一原则：

> 辅助 Codex 基于蓝湖设计文稿进行代码还原。

原因：

- 多 Agent 协作中，口头建议如果不写入仓库，很容易在不同客户端之间丢失。
- 项目可能出现很多“看起来有用”的扩展，但只有服务代码还原主线的建议才应进入路线。
- 将有效建议沉淀到 `ROADMAP/HANDOFF/DECISIONS/STATUS`，能让后续 Agent 拉取后继续沿着同一主线工作。

影响：

- 新建议不直接进入实现，先判断是否提升蓝湖读取、设计理解、资源准备、还原质量或协作效率。
- 不冲突且有价值的建议必须进入对应文档。
- 与第一原则冲突的建议应明确暂缓或拒绝，避免偏离为全能蓝湖平台工具。

## 2026-07-01：E2E 缺口优先转化为上下文可消费性补强

决定：B01 首轮页面还原 E2E 后，先补强 context 可消费性，再继续扩展更重的单图详情、切图和 tokens。

原因：

- E2E 证明当前产物已经能支撑低保真页面结构还原，但 Agent 消费链路还不够稳定。
- 当前链接的 `image_id` 指向 Axure 文档而不是 B01 画板，说明还原目标需要显式聚焦参数。
- Codex MCP 长进程可能继续使用旧构建，缺少 schema/capability 信息会让 Agent 难以及时发现产物能力不匹配。
- 图片真实像素尺寸与 API 画板尺寸存在 4 倍缩放关系，若不写入 context，会影响布局和切图判断。

影响：

- V1.1.1 优先加入 context schema/capability、目标画板聚焦、图片尺寸映射和 MCP 重启提示。
- V1.2 单图详情仍然重要，但应建立在 Agent 能稳定识别目标画板和 context 能力的基础上。
- E2E 报告成为后续优先级依据，避免凭主观感觉扩展功能。

## 2026-07-01：支持页面级与组件级代码还原

决定：后续能力不只服务整页还原，也要支持从完整蓝湖设计图中聚焦一个局部组件进行代码还原。

原因：

- 真实业务中，设计稿常常以整页形式交付，但开发任务可能只是实现底部输入栏、卡片、弹窗、CTA 或某个局部模块。
- 如果 MCP 只能输出整页 context，Agent 仍需要人工判断组件边界，效率和稳定性都会下降。
- 组件级目标仍然服务第一原则：辅助 Codex 基于蓝湖设计文稿进行代码还原。

影响：

- V1.1.1 需要预留 `targetRegion`、`targetDescription` 等组件目标参数。
- V1.2 如果能获得图层数据，应继续升级到 `targetLayerId` 或类似的精确图层选择。
- 输出中必须明确组件信息来自人工目标描述、坐标裁剪还是蓝湖图层数据，避免把推断伪装成官方结构化数据。
