# B01 页面还原 E2E 验证报告

## 验证目标

验证当前蓝湖 MCP 产物是否能辅助 Codex 在不重新打开蓝湖网页的情况下，还原 `B01留资页-用户未登录（有搜索词）`。

## 验证输入

- 蓝湖链接：`product` 链接，包含 `pid/tid/image_id/docType=axure/pageId/parentId`。
- MCP 工具：`lanhu_get_design_context`，`includeImages=true`。
- 生成时间：2026-07-01。
- 生成结果：项目 `3.3.1落地页`，读取到 8 个画板，下载 8 张缩略图。
- 目标画板：`B01留资页-用户未登录（有搜索词）`。
- 本地验证页：`work/e2e-b01-playground/`，该目录被 `.gitignore` 忽略，不提交蓝湖缩略图或还原页面资产。

## 验证过程

1. 使用 Codex MCP 中的 `lanhu_get_design_context` 重新生成 context。
2. 从生成的画板列表中人工选择 B01 缩略图。
3. 基于 `context.md/context.json` 和 B01 缩略图，创建本地静态 playground。
4. 使用本地浏览器打开 `http://127.0.0.1:4187/`，将还原页与蓝湖缩略图并排对照。
5. 检查页面结构、主要文案、视觉层级、底部输入区和可复用资产缺口。

## 验证结论

当前 MCP 产物已经可以帮助 Codex 识别页面类型、业务流程和主视觉结构，足以生成一个可运行的低保真页面还原样例。

但它还不能稳定支撑高质量页面还原，原因是当前产物主要是项目画板列表和整页缩略图，缺少图层、切图、文本、样式 token、资源分类和目标画板聚焦信息。

## 已验证可行项

- 可以通过画板名称识别 B01 是留资/咨询线索收集页。
- 可以通过缩略图还原页面大结构：顶部导航、律师服务头图、咨询分类、聊天对话流、底部快捷入口和输入栏。
- 可以在不打开蓝湖网页的情况下，用本地缩略图完成一次静态页面还原验证。
- 当前页面还原缺口可以被清晰拆分为后续 MCP 能力项。

## 发现的缺口

- 当前 Codex MCP 进程生成的 `context.md/context.json` 未包含源码中已有的 `restoration` 字段，判断为运行中 MCP 进程仍使用旧构建或未重启；这会直接影响另一端 Agent 消费最新能力。
- 当前链接中的 `image_id` 指向 Axure 文档 `H5落地页+v3.3.1`，不是 B01 画板 ID；MCP 不能自动把还原目标聚焦到 B01，只能由 Agent 人工从画板名称选择。
- `context.json` 没有记录下载后图片真实像素尺寸和 API 画板尺寸之间的缩放比例；B01 API 尺寸为 `187.5x544`，本地缩略图实际为 `750x2176`，实现时需要人工推断 4 倍缩放关系。
- 缺少图层树和文字节点，Codex 只能从缩略图读出可见文案，不能稳定获得字体、字号、行高、颜色、圆角、间距等实现 token。
- 缺少切图资源，律师人物图、头像、分类图标、底部图标只能用 CSS 占位或从整页缩略图裁剪，无法形成干净可复用资产。
- 缺少资源语义分类，无法区分 `hero人物图`、`助手头像`、`用户头像`、`分类图标`、`底部快捷入口图标` 等。
- 当前 `docType=axure` 只返回 warning，尚未解析 Axure/PRD 中可能存在的流程或交互说明。

## 下一步建议

按第一原则“辅助 Codex 基于蓝湖设计文稿进行代码还原”评估后，建议先做 V1.1.1 上下文可消费性补强，再进入 V1.2 单图详情增强：

1. 增加 context schema/capability 信息，让 Agent 能判断当前 MCP 产物是否包含 `restoration`，并在 MCP 更新后提示需要重启服务。
2. 为 `lanhu_get_design_context` 增加目标画板聚焦能力，例如 `targetImageId` 或 `targetImageName`，支持在 Axure 文档链接下明确选择 B01。
3. 在 `context.json` 中记录下载图片真实像素尺寸、API 尺寸和推断缩放比例。
4. 扩展单图详情 API 探查，优先寻找图层、文本、标注和切图入口。
5. 后续再推进切图资源、Design Tokens 和可复制还原 Prompt。
